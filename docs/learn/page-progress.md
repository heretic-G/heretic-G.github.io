如果实现一个页面加载进度条 纯js (这是个人思考和一些测试)

最开始看到这个需求

* 第一个就是知道哪些资源加载了
* 第二就是这些资源有多大
* 第三什么时候加载完了

这是我最开始的想法  然后就想到了`MutationObserver`最开始的想法就是利用他抓住img script video 等会加载资源的dom 这样我就知道了谁什么时候开始加载了

然后利用`performance.getEntries` 能够过滤出加载完的 这个其实也可以进行监听 然后加载完其实我知道大小的

不过到了这里有个资源大小获取的问题 就是 img这类我就算知道啥时候开始其实这时候我是拿不到资源大小的 然后我又发现 如果是css里面加载的资源 我可能抓取不到 js中的也可能抓取不到 等等还可能存在我没思考到的情况

这里我就开始放弃了`MutationObserver`，想到了`serviceWorker`因为它里面可以拦截fetch做缓存 同样的我也可以拦截去知道谁什么时候开始的 这里的好处是所有的请求都能拿到，`serviceWorker` 他有自己的生命周期 又不能操作外面的`performance `我这里使用`BroadcastChannel`做广播 不过因为`serviceWorker`有自己的周期所以这里逻辑稍微复杂点

大体流程
* `serviceWorker`抓到请求会记录下来和抓取的时间(这里会剔除head请求) 然后出发数组发送上去(这里没有确认回传回来不会清除)
* 页面加载后 `serviceWorker`注册后 开启监听`serviceWorker`发送上来的URL 发送一个当前页面加载时间
* `serviceWorker`  拿到时间过滤自己的map 剔除非本次页面加载记录的URL 然后发个已发送的URL数组到页面
* 页面拿到URL 发送fetch head 获取资源大小 会发送确认数组到 `serviceWorker`  `serviceWorker`会把确认的url 剔除

实现到这里其实就发现一些无法使用在生产的原因
* `serviceWorker`  注册比较慢(也可能是我第一次 不会使用导致的)
* 现在网络中 其实传输时延才是大头 大多数资源其实压缩后head 回来和资源回来都差不多了 还有就是通信需要时间这里也可能是个问题
* 页面资源少的 没有使用价值
* 第一次`serviceWorker`需要首次注册所以会很慢 第二次`serviceWorker`虽然能和页面一块处理 但是这时候大部分都是缓存….
* 现在页面异步加载资源多 需要手动设置一个阈值判断不在收录加载资源 这里不能无限等待

如果不使用编译时抓取一些信息的话 其实资源大小并不适合 除非是一个特定场景的页面可能能使用
利用资源个数或者大小在异步流程中都会有当前80%后可能一个资源异步引入很多资源加载这时候需要有两个进度一个展示进度 一个实际进度 这里实际小于展示 那展示就不能再更新 需要等实际进度上来再增加 然后全部加载后这里只能是90% 假设500ms 无新资源加载在到100%  如果是全都是同步资源的页面那其实就不需要这些逻辑
